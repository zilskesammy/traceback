Zwei Ansätze für das einheitliche Datenmodell

Wie werden Changelog-Daten und Traceback-Struktur zusammengeführt?

A

Flaches Modell — Item + Entry + CodeChange

Epic/Feature/Task werden durch ein einziges Item-Modell ersetzt. Jedes Item kann Einträge (Entries) und Codeänderungen haben. Keine Hierarchie — Gruppierung über Tags oder parentId.

Pro

  • Direkte 1:1 Abbildung der Changelog-Struktur
  • Einfache Abfragen
  • Klar für Agents: ein Modell

Con

  • Keine native Hierarchie (Epic→Feature→Task)
  • Spätere Planung braucht parentId-Logik
B

Hierarchisches Modell — Feature + Entry + CodeChange

Bestehende Hierarchie bleibt als Feature (ersetzt Epic+Feature+Task) mit optionalem parentId. Changelog-Features landen als Feature-Einträge, Entries werden angehängt.

Pro

  • Hierarchie für spätere Planung vorbereitet
  • Epics/Tasks über type+parentId möglich
  • Flexibler für Claude→UI→Execute-Workflow

Con

  • self-referential FK etwas komplexer
Bestehende Modelle bleiben in beiden Ansätzen erhalten
Project ✓ User ✓ Agent ✓ SessionStep ✓ LinkedPR ✓ Commit ✓ ApiKey ✓ Epic ✗ ersetzt Feature ✗ ersetzt Task ✗ ersetzt TaskComment ✗ in Entry aufgegangen
